1. The Mathematical Engines (The Attractors)
These are the core algorithms. In a shader context, you'll typically run these equations iteratively over thousands of particles or use them to warp a 2D coordinate space.
Clifford Attractors (The Woven Silk)
The Math: $x_{n+1} = \sin(a y_n) + c \cos(a x_n)$
$y_{n+1} = \sin(b x_n) + d \cos(b y_n)$
The Art: Clifford attractors operate in 2D and are incredibly sensitive to their $a, b, c, d$ parameters. Aesthetically, they render as gossamer, folded ribbons of light or dense, moiré-like interference patterns.
Peter de Jong Attractors (The Bioluminescent Web)
The Math:
$x_{n+1} = \sin(a y_n) - \cos(b x_n)$
$y_{n+1} = \sin(c x_n) - \cos(d y_n)$
The Art: Similar to Clifford but with a softer, more sweeping curvature. When rendered with low opacity and high particle counts, they look like glowing deep-sea jellyfish or intricate orbital maps.
Aizawa Attractor (The 3D Sphere Vortex)
The Math: 3D differential equations creating a sphere with a tube-like structure through its z-axis.
The Art: If you are raymarching or doing 3D particle projection, Aizawa is the holy grail. It looks like a swirling cosmic apple or a contained black hole. Rotating the camera around an Aizawa attractor while mapping velocity to color is visually explosive.
Thomas' Cyclically Symmetric Attractor (The Liquid Crystal)
The Math: A 3D system that lacks any dissipation, meaning it doesn't shrink down to a single point or line but forms an infinitely complex, interconnected 3D lattice.
The Art: Renders as a 3D labyrinth. It’s perfect for creating volumetric, glowing, maze-like structures that feel like alien geometry.

2. The GLSL Rendering Pipeline (Making it Art)
Calculating the math is only 10% of the visual splash. The rendering pipeline is where it becomes art.
Framebuffer Object (FBO) Ping-Ponging: Instead of clearing the canvas every frame, feed the previous frame's output back into the current frame with a slight fade (e.g., multiplying the previous frame by 0.95). This creates endless, glowing ghost trails that visualize the history of the chaos.
Additive Blending: Set the webgl blend function to additive (gl.blendFunc(gl.SRC_ALPHA, gl.ONE)). When thousands of faint particles overlap at the dense nodes of the attractor, the light accumulates into blazing, blown-out hot spots.
Velocity-to-Color Mapping: Calculate the speed of the particle (the distance between its previous position and current position) and map that float value to a color gradient. Slow particles might be deep, bruising indigos, while fast-whipping particles ignite into atomic greens and blinding yellows.
Domain Warping: Take the output of the strange attractor and feed it into a Perlin or Simplex noise function before rendering to the screen. It breaks the perfect mathematical symmetry just enough to make it feel organic and unpredictable.

3. Aesthetic Post-Processing & Textures
To pull this away from a dry, scientific Mathematica plot, the final output needs heavy atmospheric styling.
Shoegaze Smear (Temporal Blur): Combine the FBO feedback with a slight radial blur from the center of the screen. The math will look like it's dissolving into a heavy, dreamlike haze.
CRT Phosphor Glow & VHS Tracking: Apply a subtle scanline overlay and a slight RGB channel separation (chromatic aberration) at the edges of the canvas. This grounds the infinite mathematical complexity into the warm, imperfect artifacting of vintage media.
Maximalist Chromatics: Ditch standard scientific heatmaps. Build gradient palettes using extreme, hyper-saturated tones—think electric magentas, cyan plasma, deep ultraviolet voids, and radioactive greens. The goal is a tonal punch that feels nostalgic but aggressively futuristic.

1. The Attractor Math Snippets (src/shaders/attractors/)

Clifford Attractor (clifford.glsl)
OpenGL Shading Language
// A, B, C, D are typically uniforms driven by JS, 
// usually ranging between -3.0 and 3.0 for best visual results.
vec2 clifford(vec2 pos, float a, float b, float c, float d) {
    float nextX = sin(a * pos.y) + c * cos(a * pos.x);
    float nextY = sin(b * pos.x) + d * cos(b * pos.y);
    return vec2(nextX, nextY);
}


Peter de Jong Attractor (de-jong.glsl)
OpenGL Shading Language
// Creates beautiful, glowing bioluminescent webs. 
// Parameters A, B, C, D usually between -3.0 and 3.0.
vec2 deJong(vec2 pos, float a, float b, float c, float d) {
    float nextX = sin(a * pos.y) - cos(b * pos.x);
    float nextY = sin(c * pos.x) - cos(d * pos.y);
    return vec2(nextX, nextY);
}


Aizawa Attractor (3D) (aizawa.glsl)
OpenGL Shading Language
// 3D Sphere Vortex. Needs a small timestep (dt) for numerical stability.
// Default params: a=0.95, b=0.7, c=0.6, d=3.5, e=0.25, f=0.1
vec3 aizawa(vec3 pos, float a, float b, float c, float d, float e, float f, float dt) {
    float x = pos.x; float y = pos.y; float z = pos.z;
    
    float dx = (z - b) * x - d * y;
    float dy = d * x + (z - b) * y;
    float dz = c + a * z - (z * z * z) / 3.0 - (x * x + y * y) * (1.0 + e * z) + f * z * x * x * x;
    
    return pos + vec3(dx, dy, dz) * dt;
}



2. The GPGPU Compute Shader Logic (src/shaders/compute.frag)
To get millions of particles running at 60fps, you can't calculate them in JavaScript. You have to store their positions in a texture and update them on the GPU.
Give Copilot this conceptual logic to build the compute shader:
OpenGL Shading Language
uniform sampler2D u_particleData; // RGBA texture: RG = Position, BA = Velocity
uniform vec4 u_params; // a, b, c, d for the attractor
uniform float u_time;
uniform float u_speed; // Controls how fast particles step through the math

void main() {
    vec2 uv = gl_FragCoord.xy / resolution.xy;
    vec4 data = texture2D(u_particleData, uv);
    
    vec2 currentPos = data.xy;
    vec2 currentVel = data.zw;
    
    // 1. Calculate next position using chosen attractor
    vec2 targetPos = clifford(currentPos, u_params.x, u_params.y, u_params.z, u_params.w);
    
    // 2. Interpolate for smooth motion instead of instantly jumping
    vec2 nextPos = mix(currentPos, targetPos, u_speed);
    
    // 3. Calculate new velocity for color mapping in the render pass
    vec2 newVel = nextPos - currentPos;
    
    // Write data back to the FBO
    gl_FragColor = vec4(nextPos, newVel);
}



3. Velocity-to-Color Mapping (src/shaders/render.frag)
This is where the maximalist visual pop happens. The faster a particle is whipping around the attractor node, the brighter and more shifted its color should be.
OpenGL Shading Language
uniform sampler2D u_particleData;
uniform sampler2D u_colorRamp; // A 1D texture containing your hyper-saturated palette

void main() {
    // Note: gl_PointCoord is used to draw the particle as a soft circle
    vec2 uv = gl_FragCoord.xy / resolution.xy;
    vec4 data = texture2D(u_particleData, uv);
    
    vec2 velocity = data.zw;
    float speed = length(velocity);
    
    // Map speed to a 0.0 -> 1.0 range (adjust multiplier based on attractor scale)
    float normalizedSpeed = clamp(speed * 5.0, 0.0, 1.0);
    
    // Sample your vivid palette based on how fast the particle is moving
    vec3 baseColor = texture2D(u_colorRamp, vec2(normalizedSpeed, 0.5)).rgb;
    
    // Shape the particle into a soft, glowing dot rather than a harsh square
    float dist = length(gl_PointCoord - vec2(0.5));
    float alpha = smoothstep(0.5, 0.1, dist);
    
    // Low opacity is CRITICAL for additive blending to look like glowing smoke
    gl_FragColor = vec4(baseColor, alpha * 0.05); 
}

5. The Aesthetic Finisher (src/shaders/post-process.frag)
This shader takes the blazing, additive-blended particle output and grounds it with analog artifacts. Copilot can use this to build the final pass in your rendering pipeline.
OpenGL Shading Language
uniform sampler2D u_tDiffuse; // The raw rendered particles from the FBO
uniform float u_time;
varying vec2 vUv;

void main() {
    vec2 uv = vUv;
    
    // 1. Chromatic Aberration (RGB Shift)
    // Separates the color channels slightly at the edges for a lens-distortion feel
    float shift = 0.003 + (sin(u_time * 2.0) * 0.001); // Slight pulse
    float r = texture2D(u_tDiffuse, uv + vec2(shift, 0.0)).r;
    float g = texture2D(u_tDiffuse, uv).g;
    float b = texture2D(u_tDiffuse, uv - vec2(shift, 0.0)).b;
    vec3 color = vec3(r, g, b);
    
    // 2. CRT Phosphor Scanlines
    // Adds a subtle horizontal frequency interference
    float scanline = sin(uv.y * 800.0 * 3.14159) * 0.04;
    color -= scanline;
    
    // 3. Vignette (Edge Darkening)
    // Focuses the eye on the center attractor nodes
    float dist = distance(uv, vec2(0.5));
    color *= smoothstep(0.85, 0.3, dist);
    
    // 4. Shoegaze Smear (Temporal Decay)
    // Note: The actual "smear" happens in the FBO ping-pong, 
    // but we ensure the output alpha supports it here.
    gl_FragColor = vec4(color, 1.0);
}

// Function to generate a 1x256 pixel texture ramp from a hex palette
export function createNeonColorRamp(gl) {
    // A tonal guide using intense, hyper-saturated contrasts
    // Maps from slow (dark/violet) to fast (blazing cyan/yellow)
    const hexPalette = [
        "#0a001a", // Deep void purple
        "#ff007f", // Electric magenta
        "#9d00ff", // Intense violet
        "#00ffff", // Neon cyan
        "#fff700"  // Blinding yellow
    ];
    
    const canvas = document.createElement('canvas');
    canvas.width = 256;
    canvas.height = 1;
    const ctx = canvas.getContext('2d');
    
    // Create a smooth linear gradient across the 256 pixels
    const gradient = ctx.createLinearGradient(0, 0, 256, 0);
    hexPalette.forEach((hex, index) => {
        gradient.addColorStop(index / (hexPalette.length - 1), hex);
    });
    
    ctx.fillStyle = gradient;
    ctx.fillRect(0, 0, 256, 1);
    
    // Extract pixel data to send to the WebGL texture
    const imgData = ctx.getImageData(0, 0, 256, 1).data;
    
    // Copilot instruction: Bind 'imgData' to a gl.TEXTURE_2D
    // with gl.TEXTURE_MIN_FILTER = gl.LINEAR
    return imgData; 
}
